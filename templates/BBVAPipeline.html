Use Case: Predictive model for early debt resolution, BBVA Bank

This is a predictive model created by the BBVA AI Factory division of BBVA Bank with the aim of improving early debt management and personalizing it according to each case.

Problem Description

When a customer enters a delinquency state due to financial difficulties, both the customer and the bank are affected. The customer suffers penalties and the bank faces the risk of financial loss. The objective is to identify these situations early and categorize them in order to provide appropriate solutions for each case.

Type of Machine Learning

This model is a based on a machine learning pipeline that uses supervised learning because it uses historical data, where the outcome whether the customer pays or not is already known.

Therefore, it learns patterns from this labeled data to make predictions about new customers.

It is a classification model because it assigns each customer a risk category, allowing the bank to decide on the appropriate recovery strategy. This customer classification is performed as follows:

Classification and Types of models according to the client's situation

They are divided into 4 models according to the delinquency status.

This segmentation makes it possible to adapt recovery strategies according to the customer profile.

Data

Input Data

The input data used by the model consists of data labeled according to the different factors that can influence payment compliance. These are divided into the following variables:

  • Sociodemographic data (age, income, employment status)
  • Debt status
  • Payment behavior
  • Financial history

Expected Output

The model is designed to produce data that allows the bank to determine which model is appropriate for each client.

Thse outputs allow the bank to make data-driven decisions and apply personalized recovery strategies. They are divided into:

  • Probability of repayment
  • Risk classification
  • Recommended actions

Structure management

1. ETL Dataset Construction

The table is created through an extract, transform, and load (ETL) process using the input data as variables.

2. Data Split

The data is separated into:

  • Training
  • Validation
  • Test
The latter acts as a temporal holdout set to review the model's behavior.

3. Variable Selection and Control

Using the GRMlab tool, variables are preselected. Afterwards, unstable variables (data drift) are identified and removed. Finally, using the Mercury tool, model overfitting is controlled, preventing performance deterioration.

4. Modeling

Three types of models are built:

  • Baseline Model: integrates traditional methodologies that allow referencing the minimum expected performance.

  • Complex non-productionizable model: AutoGluon is used to generate code pipelines automatically from the data, producing an efficient and high-performing model, but not suitable for production integration due to its lack of transparency and interpretability.

  • Production candidate model: represents a balance between the type of algorithm, variable selection, and hyperparameter optimization. It also allows the integration of algorithm lists and processing through time-series cross-validation, ensuring stable performance.

At this stage two key concepts appear:

  • Feature Selection: techniques such as Permutation Feature Importance and Recursive Feature Elimination are used to discard redundant variables.

  • Hyperparameter Optimization using the Optuna framework to maximize AUC, minimize overfitting, and apply time-series cross-validation.

5. Model Evaluation

To determine the effectiveness of the model, the following tests are carried out:

  1. Performance measurement using metrics such as the Gini index, recall, and the confusion matrix.
  2. Evaluation by segments such as product type or financial history.
  3. Stability analysis and deviation index.
  4. Using Optimal Binning to discretize the model score, which allows specific actions depending on the risk level.
  5. Variable impact analysis using the SHAP library.
  6. Bias absence verification.

6. Deployment and Monitoring

Once the tests are passed, the model is implemented in production and monitored for:

  • Performance
  • Stability
  • Retraining if necessary

Real world relevance

This case is relevant in real-world situations because, in addition to being successful, it allows banks to reduce losses and improve customer management based on data.

Furthermore, by being scalable and combining performance with interpretability, it adapts dynamically to changes in the financial market.
